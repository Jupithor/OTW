Welcome to OverTheWire!

If you find any problems, please report them to Steven or morla on
irc.overthewire.org.

--[ Playing the games ]--

  This machine might hold several wargames.
  If you are playing "somegame", then:

    * USERNAMES are somegame0, somegame1, ...
    * Most LEVELS are stored in /somegame/.
    * PASSWORDS for each level are stored in /etc/somegame_pass/.

  Write-access to homedirectories is disabled. It is advised to create a
  working directory with a hard-to-guess name in /tmp/.  You can use the
  command "mktemp -d" in order to generate a random and hard to guess
  directory in /tmp/.  Read-access to both /tmp/ and /proc/ is disabled
  so that users can not snoop on eachother. Files and directories with
  easily guessable or short names will be periodically deleted!

  Please play nice:

    * don't leave orphan processes running
    * don't leave exploit-files laying around
    * don't annoy other players
    * don't post passwords or spoilers
    * again, DONT POST SPOILERS!
      This includes writeups of your solution on your blog or website!

--[ Tips ]--

  This machine has a 64bit processor and many security-features enabled
  by default, although ASLR has been switched off.  The following
  compiler flags might be interesting:

    -m32                    compile for 32bit
    -fno-stack-protector    disable ProPolice
    -Wl,-z,norelro          disable relro

  In addition, the execstack tool can be used to flag the stack as
  executable on ELF binaries.

  Finally, network-access is limited for most levels by a local
  firewall.

       More detail on the following is available from the [1mGit[0m [1mconcepts[0m [1mchapter[0m [1mof[0m [1mthe[0m [1muser-manual[0m[2][m
       and [1mgitcore-tutorial[0m(7).[m
[m
       A Git project normally consists of a working directory with a ".git" subdirectory at the top[m
       level. The .git directory contains, among other things, a compressed object database[m
       representing the complete history of the project, an "index" file which links that history to[m
       the current contents of the working tree, and named pointers into that history such as tags and[m
       branch heads.[m
[m
       The object database contains objects of three main types: blobs, which hold file data; trees,[m
       which point to blobs and other trees to build up directory hierarchies; and commits, which each[m
       reference a single tree and some number of parent commits.[m
[m
       The commit, equivalent to what other systems call a "changeset" or "version", represents a step[m
       in the projectâ€™s history, and each parent represents an immediately preceding step. Commits[m
       with more than one parent represent merges of independent lines of development.[m
[m
       All objects are named by the SHA-1 hash of their contents, normally written as a string of 40[m
       hex digits. Such names are globally unique. The entire history leading up to a commit can be[m
       vouched for by signing just that commit. A fourth object type, the tag, is provided for this[m
       purpose.[m
[m
       When first created, objects are stored in individual files, but for efficiency may later be[m
       compressed together into "pack files".[m
[m
       Named pointers called refs mark interesting points in history. A ref may contain the SHA-1 name[m
       of an object or the name of another ref. Refs with names beginning [1mref/head/[0m contain the SHA-1[m
       name of the most recent commit (or "head") of a branch under development. SHA-1 names of tags[m
       of interest are stored under [1mref/tags/[0m. A special ref named [1mHEAD[0m contains the name of the[m
       currently checked-out branch.[m
[m
       The index file is initialized with a list of all paths and, for each path, a blob object and a[m
       set of attributes. The blob object represents the contents of the file as of the head of the[m
       current branch. The attributes (last modified time, size, etc.) are taken from the[m
       corresponding file in the working tree. Subsequent changes to the working tree can be found by[m
       comparing these attributes. The index may be updated with new content, and new commits may be[m
       created from the content stored in the index.[m
[m
       The index is also capable of storing multiple entries (called "stages") for a given pathname.[m
       These stages are used to hold the various unmerged version of a file when a merge is in[m
       progress.[m
